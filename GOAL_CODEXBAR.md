# GOAL: 完整对齐 CodexBar 用量能力

> 本文件是 Conductor 对齐 CodexBar 的长期目标与执行 runbook。
> 目标不是“看起来像 CodexBar”，而是把 CodexBar 已有、Conductor 缺失或半成品的用量能力逐项转写过来，直到 provider 覆盖、数据源、账户体系、成本历史、CLI/API、设置与 UI 细节都达到同等能力。

## 总目标

把 CodexBar 的全部用量功能转写到 Conductor：

- CodexBar 有、Conductor 没有的功能，要补齐。
- Conductor 有但能力弱、数据源少、UI 粗糙、口径不准的功能，要按 CodexBar 的真实逻辑重写或补足。
- 对齐以 CodexBar 源码和文档为准，不凭印象，不做“差不多”。
- 能直接搬逻辑就尽量转写逻辑；不要为了兼容旧半成品保留分叉、旧路径或影子实现。

完成时，Conductor 的用量模块应能作为 CodexBar 的上位替代：provider 覆盖一致，核心数据口径一致，CLI/脚本可用，UI 能展示完整账号、额度、成本、历史、错误和修复路径。

## 重新部署规则（2026-06-19）

本目标是长期 runbook，不再作为单次无界长任务执行。每一轮只选择一个可验证缺口，先写清验收 gate，再改代码；完成该缺口、遇到阻塞或验证失败后立即停下来汇报，不继续扩散到其它大项。

- 每轮最多推进一个主题，例如“MiMo auth redirect parity”“设置页 capability 动态化”“90 天用量抽屉错位”。
- 不用 `16/33` 这种大项数字冒充进度；大项只有在当前证据足以覆盖整项时才改成完成。
- `🔄 初版已补` 项必须继续拆小验收点，记录具体已闭环内容和仍缺内容。
- 开始改代码前先确认 dirty diff，标出未验证改动；上一轮中断留下的改动必须先验证、补完或撤掉。
- 每轮必要 gate 优先级：相关专项 smoke/audit、`swift build --product conductorctl`、`swift build -c debug --product ConductorApp`；`swift test` 若仍因本机缺 `XCTest` 失败，只记录工具链问题，不把它算作代码失败。
- UI 改动必须至少通过构建和相关审计；涉及错位/重叠时需要运行或截图验证。
- 不再连续长时间自动追大目标；一组改动和 gate 跑完就停。

## 执行原则

1. **真实对齐，不做兼容壳**
   - 如果现有实现是半成品，直接重写或替换。
   - 不保留旧数据结构、旧路径、旧 UI 分支，除非当前用户数据必须迁移且迁移本身是目标的一部分。

2. **尽量转写 CodexBar 逻辑**
   - provider fetcher、parser、fallback order、错误语义、重置时间、百分比口径、成本估算，优先照 CodexBar 实现。
   - 只有在 Conductor 架构确实不同的时候，才做最小必要适配。

3. **统一核心模型**
   - provider 能力、metadata、fetch plan、source mode、token account、cost history 不要散落在 UI 里。
   - 纯逻辑放 `ConductorCore`，UI 只负责展示和触发。

4. **人和机器都好用**
   - GUI 要能完整管理 provider、凭证、刷新、错误和历史。
   - CLI/socket 要能读取同一套状态、触发同一套刷新、输出结构化 JSON。

5. **验收以当前 CodexBar 为准**
   - 每次实现前重新生成差距表。
   - 实现后用差距表逐项划掉，不能靠“我觉得差不多”。

## 固定流程

每次恢复这个目标，按顺序执行：

1. **刷新 CodexBar 源码**
   - 拉 `/tmp/codexbar-audit`。
   - 抽取 provider registry、CLI 命令、关键 docs/changelog。

2. **生成差距表**
   - provider 覆盖差距：先跑 `python3 Scripts/audit-codexbar-providers.py --codexbar /tmp/codexbar-audit`，确认 provider registry、sourceModes 与 metadata 自动审计通过，再人工看 provider 专属行为差距。
   - 每个 provider 的数据源差距：api/oauth/web/cookie/cli/local。
   - 成本历史与本地扫描差距。
   - CLI/API 差距。
   - UI/设置/账号/错误态差距。

3. **选择第一个未完成差距**
   - 优先级：缺 provider -> 核心 Codex/Claude 成本口径 -> CLI usage/cost/serve -> UI/设置深度 -> 通知/widget/status 细节。
   - 一次只做一个闭环，不并行开一堆半成品。

4. **转写实现**
   - Core parser/fetcher 先落地。
   - 再接 catalog/config/credentials。
   - 最后接 UI 与 CLI。
   - 删除被替代的旧逻辑。

5. **补测试**
   - parser/fetcher 用 CodexBar 测试思路补样例测试。
   - 错误、空数据、未登录、401/403、分页、重置时间、金额/积分单位都要覆盖。
   - UI 变更至少跑构建，关键布局用截图确认。

6. **验收**
   - `swift build` 必须通过。
   - 能跑的单测要跑；当前本机 XCTest 不可用时要明确记录原因。
   - provider 差距表更新。
   - 文案走 `L(...)`，英文表不漏。

7. **提交**
   - 一个功能一个 commit。
   - commit message 写清楚对齐了 CodexBar 哪个 provider/能力。
